# Privacy and Data Handling

This document describes how Jarvis handles project data when a local Jarvis client is connected to the Jarvis server.

Jarvis is designed for working with files that remain on the user's local machine. A user runs the local client, chooses a workspace directory, and gives an agent the temporary session key displayed by the client. The server uses that key to route agent commands to the connected client.

## What Jarvis Can Access

The local client can access files and folders inside the workspace selected by the user. Project-relative path checks are used to prevent access outside the selected workspace/project root.

Depending on the command approved by the user, Jarvis may:

- list projects, directories, and file metadata;
- search file names or file contents;
- read one or more files;
- write or patch files;
- run bounded local commands;
- inspect git status or diffs;
- create local git commits;
- start, inspect, or cancel local jobs.

## Data Sent Through the Server

The server routes command requests from the agent to the connected client and routes command responses back to the agent. Command responses may include project file contents, search results, command output, git output, error messages, file paths, and other command results.

Jarvis does not intentionally persist project file contents on the server. File contents and command output may still pass through server memory while a command is being handled and may be visible to normal hosting, diagnostics, reverse-proxy, or error-logging infrastructure depending on deployment configuration.

## Server-Side State

The server keeps temporary in-memory state needed to operate the service, including connected client/session routing information and pending command response tracking.

Agent endpoints require an API key. Client sessions use temporary keys generated by the local client. These keys are used to route commands and should be treated as sensitive while the client is running.

## Client-Side Audit Data

The local client keeps an in-memory audit log for sensitive command categories such as file writes, patches, process execution, git commits, job starts, and job cancellation.

Audit entries may include:

- timestamp;
- command name;
- project name;
- permission level;
- affected paths;
- executable name and arguments;
- result summary.

The audit log is held in client memory and is not intended to contain full project file contents.

## User Control and Permissions

Read-only commands can inspect project structure and contents exposed by the selected workspace. Mutating, process-execution, and version-control-write commands require local approval from the user before they run.

Users can stop access by closing the local client, quitting the session, or refusing permission prompts.

## Data Retention

Jarvis is primarily a routing service between the agent and the local client. The project does not intentionally store project file contents on the server.

Operational logs, hosting logs, reverse-proxy logs, crash reports, or diagnostics may be retained according to the configuration of the environment where the server is deployed.

## Data Sharing

Jarvis sends command results to the agent that issued the command for the active session. Project data is not intentionally shared with third parties by Jarvis, except as necessary to operate the service, comply with law, or support the hosting environment.

## Security Notes

Users should only run the client for workspaces they are comfortable exposing to the agent. Users should treat session keys as sensitive and avoid sharing them with anyone who should not be able to send commands to the client.
